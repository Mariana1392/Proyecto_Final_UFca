🔒1. Seguridad

S-01 CRÍTICA SEGURIDAD Archivo .env incluido en el proyecto / backups
.env

El archivo .env con la URL y la anon key de Supabase existe dentro del directorio del proyecto. Al compartir el proyecto en ZIP (como el backup actual) o subir a GitHub sin .gitignore correcto, las credenciales quedan expuestas públicamente.

✅ RecomendaciónAgregar .env al .gitignore. Nunca incluirlo en backups compartidos. Documentar las variables en un archivo .env.example con valores de ejemplo, no reales.
S-02 CRÍTICA SEGURIDAD Inserción de usuario + rol desde el cliente en el registro
src/components/Login.tsx — líneas 182–210

El registro llama a supabase.auth.signUp() y luego inserta manualmente en la tabla usuarios incluyendo el rol_id. Un atacante que intercepte la petición puede modificar el payload y asignarse el rol de administrador si las políticas RLS no lo impiden completamente.

✅ RecomendaciónEl INSERT en usuarios debe realizarse desde un trigger PostgreSQL (AFTER INSERT ON auth.users) con SECURITY DEFINER. El cliente nunca debe controlar el rol_id de un usuario nuevo.
S-03 CRÍTICA SEGURIDAD Cliente puede seleccionar su propio rol durante el registro
src/components/Login.tsx — líneas 199–210

El código busca el rol "usuario" en BD y lo inserta como rol_id. Si el usuario manipula la petición de red (con DevTools o Burp Suite) antes de que se ejecute el INSERT, puede reemplazar el UUID del rol "usuario" por el UUID del rol "admin".

✅ RecomendaciónUsar un trigger de BD que asigne automáticamente el rol "usuario" a todo registro nuevo. El rol solo debe cambiarse mediante funciones RPC con SECURITY DEFINER ejecutadas por el administrador.
S-04 ALTA SEGURIDAD loading: false hardcodeado — ventana de acceso sin autenticar
src/contexts/AuthContext.tsx — línea 165

El contexto inicia con loading: false sin esperar a que Supabase confirme si hay sesión activa. Existe un frame temporal donde isAuthenticated = false y loading = false simultáneamente, haciendo que los guards de ruta fallen y muestren contenido público brevemente.

✅ RecomendaciónInicializar loading: true. Cambiar a false solo después de que supabase.auth.getSession() resuelva. Mostrar un spinner global mientras se verifica la sesión.
S-05 ALTA SEGURIDAD catch {} vacío en funciones críticas de autenticación
src/contexts/AuthContext.tsx — líneas 84, 141

Los bloques catch {} en cargarPerfil y recargarPerfil absorben silenciosamente errores de BD, errores de red y tokens expirados. En cargarPerfil un fallo no llama a setU(null), dejando al usuario en estado indeterminado.

✅ RecomendaciónCapturar y loguear el error con console.error. En cargarPerfil, llamar setU(null) ante cualquier error para garantizar estado limpio y seguro.
S-06 ALTA SEGURIDAD INSERT anónimo irrestricto en solicitudes_asociados (spam)
supabase_rls_policies.sql — líneas 291–293

La política RLS permite inserción con WITH CHECK (true) a cualquier visitante. Un atacante puede automatizar el envío masivo de solicitudes de afiliación, saturando la tabla y el flujo de trabajo del administrador.

✅ RecomendaciónImplementar rate limiting en una Supabase Edge Function. Como mínimo, verificar que cedula, nombres y email no sean vacíos mediante una función RPC con validación.
S-07 ALTA SEGURIDAD RLS no cubre tablas de ventas, compras, pedidos, eventos
src/lib/api.ts — líneas 376–776 | supabase_rls_policies.sql

El archivo de API hace operaciones sobre tablas como ventas, compras, pedidos, eventos, productos, proveedores, entre otras. No existen políticas RLS definidas para estas tablas. Si RLS está habilitado, todas las queries fallan. Si está deshabilitado, cualquier usuario autenticado tiene acceso total.

✅ RecomendaciónAuditar en Supabase Dashboard el estado de RLS de cada tabla. Definir políticas explícitas para todas las tablas que usa la aplicación o deshabilitar RLS donde sea justificado y documentado.
S-08 ALTA SEGURIDAD Fallback en pagosCreditoApi que bypasea RLS y atomicidad
src/lib/api.ts — líneas 333–369

La función intenta un RPC con SECURITY DEFINER y, si falla, ejecuta un INSERT directo en pagos_credito con una actualización separada del saldo. Esto crea dos problemas: (1) bypasea RLS y las validaciones del RPC, y (2) si el INSERT es exitoso pero el UPDATE del saldo falla, los datos quedan inconsistentes.

✅ RecomendaciónEliminar el fallback completamente. Si el RPC falla, propagar el error al usuario. Las operaciones financieras deben ser atómicas en una única transacción de BD.
S-09 ALTA SEGURIDAD Lógica de negocio financiera ejecutada exclusivamente en el cliente
src/components/Creditos.tsx | src/services/businessRules.ts

El cálculo de cuotas, tablas de amortización y validaciones de reglas de negocio ocurren en el frontend. Un usuario malintencionado puede manipular los datos en la consola del navegador antes de enviarlos, enviando montos, saldos o tasas incorrectas directamente a Supabase.

✅ RecomendaciónImplementar RPCs PostgreSQL con SECURITY DEFINER para toda operación financiera crítica (registrar pago, calcular cuota, aplicar mora). El cliente solo envía los parámetros; el servidor valida y calcula.
S-10 ALTA SEGURIDAD Auditoría escrita desde el cliente — debe ser trigger de BD
src/services/businessRules.ts — líneas 447–453

Los registros de auditoría se insertan directamente desde el frontend. Esto significa que un administrador malintencionado puede omitir registrar sus acciones o falsificar entradas. La auditoría solo es confiable si ocurre server-side y es inmutable.

✅ RecomendaciónCrear triggers PostgreSQL (AFTER INSERT/UPDATE/DELETE) en las tablas críticas que escriban automáticamente en auditoria. La tabla de auditoría no debe permitir UPDATE ni DELETE.
S-11 MEDIA SEGURIDAD Checkbox "Recordarme" en Login — promesa visual sin implementación
src/components/Login.tsx — líneas 449–451

El checkbox "Recordarme" no tiene lógica asociada. La sesión siempre usa sessionStorage (se pierde al cerrar la pestaña). El checkbox genera una expectativa de usuario que no se cumple.

✅ RecomendaciónImplementar persistencia real con localStorage cuando esté marcado, o eliminar el checkbox hasta que se implemente.

🏗️2. Arquitectura

A-01 ALTA ARQUITECTURA Navegación con useState en lugar de React Router
src/App.tsx — líneas 31–179

La aplicación usa un useState<View> para navegar entre las 20+ páginas. Consecuencias: el botón "Atrás" del navegador no funciona, no existen URLs directas (no se puede compartir un enlace a un módulo específico), y toda la app vive en /.

✅ RecomendaciónMigrar a React Router v6 con rutas protegidas (<PrivateRoute>). Cada módulo debe tener su propia URL (/dashboard, /asociados, /creditos).
A-02 ALTA ARQUITECTURA userData: any como prop en 13+ componentes
src/components/Dashboard.tsx, Layout.tsx, Creditos.tsx, MiPerfil.tsx (y más)

El tipo any para userData elimina toda la protección de TypeScript. Ya se observan inconsistencias: userData.nombre ?? userData.name indica que el campo tiene dos nombres distintos dependiendo del contexto.

✅ RecomendaciónReemplazar userData?: any con el tipo AuthUser ya definido en AuthContext.tsx en todos los componentes.
A-03 ALTA ARQUITECTURA Doble fuente de verdad: user y userData en AuthContext
src/contexts/AuthContext.tsx — líneas 155–161

El contexto expone user (tipo AuthUser) y userData (objeto con campos duplicados y renombrados: name/nombre, role/rol/rol_nombre). Dos fuentes de verdad para los mismos datos inevitablemente generan inconsistencias.

✅ RecomendaciónEliminar userData del contexto y exponer solo user con el tipo AuthUser. Los componentes deben adaptarse para leer de user.
A-04 ALTA ARQUITECTURA Dead code: Dashboard-simple-backup.tsx y UserManagementContext.tsx
src/Dashboard-simple-backup.tsx | src/contexts/UserManagementContext.tsx

Archivos de respaldo y contextos no utilizados dentro del directorio src/ productivo. Inflan el bundle y confunden a futuros desarrolladores.

✅ RecomendaciónEliminar ambos archivos. Si se necesitan como referencia, guardarlos fuera del directorio src/ o en una rama separada de git.
A-05 ALTA ARQUITECTURA api.ts incluye módulos no implementados (ventas, compras, pedidos, eventos)
src/lib/api.ts — líneas 375–776

El archivo exporta APIs para módulos (ventas, compras, pedidos, eventos, productos) que no tienen vistas en App.tsx. Es código muerto que infla el bundle e introduce consultas a tablas que pueden no existir o no tener RLS.

✅ RecomendaciónSeparar api.ts por dominio (api/creditos.ts, api/ahorros.ts) y eliminar o mover a una rama los módulos no implementados.
A-06 MEDIA ARQUITECTURA Sin React.StrictMode en main.tsx
src/main.tsx

La aplicación no usa React.StrictMode, lo que impide detectar en desarrollo efectos secundarios involuntarios, uso de APIs obsoletas y llamadas duplicadas a APIs.

✅ RecomendaciónEnvolver <App /> con <React.StrictMode> en main.tsx.
A-07 MEDIA ARQUITECTURA VIEW_PERMISO incompleto — vista mi-perfil sin guard de permiso
src/lib/permissions.ts — líneas 51–65

La vista 'mi-perfil' está en el tipo View de App.tsx pero no está en el mapa VIEW_PERMISO. El guard de permisos no evalúa nada para esa ruta.

✅ RecomendaciónAgregar 'mi-perfil': [PERM.DASHBOARD, PERM.SOLICITUD_ASOCIACION] al mapa VIEW_PERMISO.
A-08 MEDIA ARQUITECTURA Vite config con aliases de versión específica frágiles
vite.config.ts — líneas 12–49

Los aliases mapean 'sonner@2.0.3': 'sonner', 'recharts@2.15.2': 'recharts', etc. Si las versiones de las dependencias cambian, los aliases dejan de funcionar silenciosamente en producción.

✅ RecomendaciónIdentificar qué dependencia importa con versión pinneada, actualizarla, y eliminar los aliases de versión del config de Vite.

🗄️3. Base de Datos

B-01 ALTA BD Columna referido_por_id ausente en migración principal de asociados
supabase_migration_completa_ufca.sql — tabla asociados

El tipo TypeScript Asociado define referido_por_id?: string, el componente Referidos.tsx hace un self-join y asociadosApi.getReferidos() filtra por esta columna. Si la columna no existe en la BD, todas las operaciones de referidos fallarán con error de PostgREST en producción. Se requiere ejecutar supabase_referidos.sql que agrega la columna.

✅ RecomendaciónEjecutar supabase_referidos.sql en Supabase SQL Editor. Verificar que la columna existe con: SELECT column_name FROM information_schema.columns WHERE table_name='asociados' AND column_name='referido_por_id';
B-02 ALTA BD Falta de índices en columnas FK críticas
supabase_migration_completa_ufca.sql — sección de índices

Columnas sin índice detectadas: creditos.asociado_id, creditos.estado (filtrada en casi todas las queries del dashboard), ahorros_permanentes.asociado_id, ahorros_voluntarios.asociado_id, usuarios.email (usada en login), usuarios.rol_id, pagos_credito.credito_id.

✅ RecomendaciónAgregar CREATE INDEX IF NOT EXISTS para todas las columnas mencionadas. Los índices en FK mejoran dramáticamente el rendimiento de JOINs y filtros frecuentes.
B-03 ALTA BD liquidaciones.detalle como JSONB para datos financieros críticos
supabase_migration_completa_ufca.sql | src/lib/supabase.ts — líneas 162–168

La columna detalle de liquidaciones es JSONB y almacena campos como estado, montos y conceptos financieros. Usar JSONB para datos financieros críticos rompe la integridad referencial, impide validaciones CHECK, dificulta queries y hace imposibles los índices eficientes en esos campos.

✅ RecomendaciónExtraer los campos críticos (estado, monto_total, tipo) a columnas tipadas con NOT NULL y CHECK constraints. El JSONB puede conservarse solo para metadatos adicionales opcionales.
B-04 ALTA BD CHECK de monto mínimo hardcodeado en BD vs configurable en tabla configuracion
supabase_migration_completa_ufca.sql — tabla pagos_ahorro_voluntario

La restricción CHECK (monto >= 50000) está hardcodeada en el schema. Si el administrador cambia el monto mínimo desde la UI (tabla configuracion), la restricción de BD no se actualiza, causando errores de inserción que el usuario no puede resolver.

✅ RecomendaciónEliminar el CHECK hardcodeado. La validación del monto mínimo debe realizarse en la función RPC que lea el valor de configuracion en tiempo de ejecución.
B-05 ALTA BD Query a tabla 'ahorros' inexistente en Dashboard stats
src/components/Dashboard.tsx — líneas 191–196

La función cargarStatsAsociado hace un join asociados(...,ahorros(monto_ahorrado,tipo)). La tabla ahorros no existe — el schema usa ahorros_permanentes y ahorros_voluntarios separadas. Esta query fallará con error de PostgREST en producción.

✅ RecomendaciónCorregir la query para usar las dos tablas correctas, igual que hace DashboardAsociado.tsx.
B-06 ALTA BD solicitudes_asociados sin restricción UNIQUE en cedula
supabase_migration_completa_ufca.sql — tabla solicitudes_asociados

A diferencia de asociados.cedula que tiene UNIQUE, la tabla de solicitudes no tiene restricción de unicidad. Un visitante puede enviar múltiples solicitudes con la misma cédula, saturando el flujo de trabajo del comité.

✅ RecomendaciónAgregar: ALTER TABLE solicitudes_asociados ADD CONSTRAINT uq_solicitudes_cedula UNIQUE(cedula) WHERE estado = 'pendiente';
B-07 MEDIA BD auditoria.detalle con tipo inconsistente (JSONB o string)
src/lib/supabase.ts — línea 211 | src/services/businessRules.ts

El tipo TypeScript declara detalle: Record<string,any> | string. En businessRules.ts se inserta como string plano. Esto provoca que algunos registros de auditoría sean JSONB y otros strings, dificultando queries y reportes.

✅ RecomendaciónNormalizar detalle siempre como JSONB con estructura fija: {"accion", "actor_id", "descripcion", "datos_anteriores", "datos_nuevos"}.
B-08 MEDIA BD Eliminación física de registros financieros (créditos y pagos)
src/lib/api.ts — líneas 276–281

creditosApi.eliminar() hace DELETE físico de créditos y sus pagos asociados. Los registros financieros nunca deben borrarse físicamente por razones de auditoría, compliance y trazabilidad.

✅ RecomendaciónEliminar la función eliminar(). La anulación lógica (anulado: true) es suficiente. Agregar un constraint de BD que impida DELETE en tablas financieras.

⚡4. Rendimiento

R-01 ALTA RENDIMIENTO Sumas de ahorros calculadas en JavaScript sobre todas las filas
src/lib/api.ts — líneas 828–887 (dashboardApi.getStats)

Las queries de ahorros traen TODAS las filas con monto_ahorrado para sumarlas en JavaScript con .reduce(). Con cientos de asociados, esto descarga miles de filas innecesariamente desde Supabase.

✅ RecomendaciónUsar agregación en BD: .select('monto_ahorrado.sum()') o crear una función RPC que devuelva los totales calculados directamente en PostgreSQL.
R-02 ALTA RENDIMIENTO asociadosApi.getAll() sin paginación — descarga toda la tabla
src/lib/api.ts — líneas 55–62 | src/components/Asociados.tsx

getAll() hace .select('*').order('nombre') sin .range() ni .limit(). Con 500+ asociados se descarga toda la tabla en memoria del cliente. La paginación y búsqueda se hacen filtrando un array local en JavaScript.

✅ RecomendaciónImplementar paginación real con .range(from, to) y .count('exact'). Los filtros de búsqueda deben aplicarse con .ilike('nombre', '%term%') en BD.
R-03 ALTA RENDIMIENTO Suscripción Realtime dispara 11 queries por cada cambio en créditos
src/components/Dashboard.tsx — líneas 52–61

La suscripción en tiempo real a creditos dispara cargarDatosGraficas() Y cargarStatsAdmin() en cada cambio, ejecutando hasta 11 queries simultáneas por cada UPDATE en la tabla.

✅ RecomendaciónConsolidar todas las queries del dashboard en una función RPC. Usar debounce (mínimo 2 segundos) para las suscripciones en tiempo real.
R-04 MEDIA RENDIMIENTO cargarSolicitudesPendientes se ejecuta en cada cambio de vista
src/components/Layout.tsx — líneas 110–124

El useEffect que carga el badge de solicitudes pendientes tiene currentView como dependencia, ejecutando una query a BD cada vez que el usuario navega a cualquier módulo.

✅ RecomendaciónEliminar currentView de las dependencias. Usar suscripción Realtime de Supabase para el badge o un intervalo de 60 segundos.
R-05 MEDIA RENDIMIENTO select('*') generalizado en múltiples APIs
src/lib/api.ts — líneas 58, 67, 132, 181, 230, 421, 462

Varias funciones usan select('*') cargando todas las columnas incluyendo campos pesados o innecesarios para el caso de uso (ej: ocupacion, direccion, motivo_anulacion en listas de asociados).

✅ RecomendaciónEspecificar solo las columnas necesarias. Para listas: columnas de presentación. Para detalle: select('*') es aceptable.
R-06 MEDIA RENDIMIENTO Clases Tailwind dinámicas no detectables en el build de producción
src/components/DashboardAsociado.tsx — líneas 339–343

Se usan interpolaciones dinámicas como hover:bg-${color}-50 y bg-${color}-100. Tailwind CSS purga clases no detectadas estáticamente, lo que puede provocar que en producción esos estilos no se generen correctamente.

✅ RecomendaciónUsar un mapa de clases estático: const colorMap = { emerald: 'bg-emerald-100 hover:bg-emerald-50', ... }.

💻5. Calidad de Código (TypeScript)

Q-01 ALTA CALIDAD Omit<any, ...> anula TypeScript completamente en la capa API
src/lib/api.ts — líneas 74, 138, 187, 236, 386, 427, 468

Omit<any, 'id' | 'created_at'> evalúa a any, por lo que no hay verificación de tipos en los parámetros de creación. Permite pasar objetos con campos incorrectos sin error en compilación.

✅ RecomendaciónUsar los tipos definidos en supabase.ts: Omit<Asociado, 'id' | 'created_at'>, etc. Para tipos no definidos, crear interfaces explícitas.
Q-02 ALTA CALIDAD Casting (data as any) generalizado en funciones críticas de auth
src/contexts/AuthContext.tsx — líneas 63–70, 120–126

El código usa (data as any) para acceder a campos de relaciones Supabase en cargarPerfil y recargarPerfil, las dos funciones más críticas del sistema.

✅ RecomendaciónDefinir una interfaz para el resultado de la query o usar el cliente Supabase tipado con tipos generados (supabase gen types typescript --project-id <id>).
Q-03 MEDIA CALIDAD handleMenuClick con 20+ condicionales else-if en cascada
src/components/Layout.tsx — líneas 192–241

La función de navegación usa una cadena de else if que debería ser un objeto de mapeo. También contiene casos muertos: id === "gestion-acceso" navega a "acceso" que no existe como vista.

✅ RecomendaciónReemplazar con const navMap: Record<string, string> = {...} y llamar onNavigate(navMap[id] ?? id). Eliminar los casos muertos.
Q-04 MEDIA CALIDAD bonificacion: 50000 hardcodeado en Referidos.tsx
src/components/Referidos.tsx — línea 60

El valor de bonificación por referido está hardcodeado. Si el administrador cambia el valor en la tabla configuracion, no se refleja en la UI.

✅ RecomendaciónCargar el valor desde configuracion con la clave correspondiente, igual que hace businessRules.ts con otros parámetros.
Q-05 MEDIA CALIDAD index.html con lang="en" — app en español
index.html — línea 3

El HTML declara lang="en" aunque toda la aplicación está en español. Esto afecta accesibilidad, lectores de pantalla y la detección automática del navegador (que ofrece traducción aunque ya está en español).

✅ RecomendaciónCambiar a <html lang="es">. Agregar metaetiquetas de descripción y, opcionalmente, una política CSP básica.
Q-06 MEDIA CALIDAD BusinessRulesEngine singleton con estado mutable y sin manejo de concurrencia
src/services/businessRules.ts — líneas 61–62, 507–509

this.config es estado mutable compartido. Si loadConfigFromDB() falla silenciosamente, el singleton mantiene valores por defecto sin notificar. Múltiples llamadas concurrentes pueden crear race conditions.

✅ RecomendaciónHacer loadConfigFromDB idempotente con un flag cargando. Exponer configCargada: boolean para que la UI pueda mostrar un aviso si la configuración no está disponible.
Q-07 BAJA CALIDAD ErrorBoundary no protege el nivel de createRoot
src/main.tsx

El ErrorBoundary está dentro de App, pero si App falla antes de montar el boundary, el error no se captura de forma amigable.

✅ RecomendaciónMover el ErrorBoundary al nivel del createRoot(...).render() en main.tsx.
Q-08 BAJA CALIDAD Links de redes sociales y footer sin funcionalidad real
src/components/Layout.tsx — líneas 551–582

Los iconos de Facebook, Twitter, Instagram y LinkedIn son href="#". Los items del footer son <span> con cursor-pointer sin onClick, decorativos sin funcionalidad.

✅ RecomendaciónConfigurar las URLs reales de redes sociales o eliminar los elementos hasta que estén disponibles.

🔍7. Hallazgos Adicionales

X-01 ALTA BD validarAfiliacionUnica verifica existencia en lugar de duplicidad
src/services/businessRules.ts — líneas 155–178

La función busca si el asociadoId ya existe activo y lo marca como violación. La lógica para prevenir duplicados debería buscar por cédula o email del solicitante, no por el mismo ID (que por definición sería diferente para cada persona nueva).

✅ RecomendaciónRevisar la lógica de negocio. La búsqueda de duplicados debe hacerse por cedula o email en la tabla asociados.
X-02 MEDIA ARQUITECTURA supabase gen types no está configurado — tipos de BD no sincronizados
package.json (ausencia de script gen:types)

Los tipos en src/lib/supabase.ts se mantienen manualmente. Si el schema de BD cambia, los tipos de TypeScript no se actualizan automáticamente, generando divergencia entre código y BD.

✅ RecomendaciónAgregar al package.json: "gen:types": "supabase gen types typescript --project-id <id> > src/lib/database.types.ts" y ejecutarlo al hacer cambios de schema.
X-03 MEDIA SEGURIDAD Sin política de Content Security Policy (CSP)
index.html

No hay cabeceras CSP ni meta tag CSP. Sin CSP, ataques XSS pueden ejecutar scripts arbitrarios en el contexto de la aplicación, con acceso al localStorage, sessionStorage y tokens de Supabase.

✅ RecomendaciónConfigurar CSP a nivel de servidor (Netlify/Vercel headers) o como meta tag. Política mínima: default-src 'self'; connect-src 'self' *.supabase.co;
X-04 BAJA CALIDAD Sin tests unitarios ni de integración
package.json — ausencia de Vitest / Jest / Cypress

No existe ningún test en el proyecto. Cambios en componentes críticos (AuthContext, businessRules, api.ts) no tienen cobertura automática, lo que hace que regresiones pasen desapercibidas.

✅ RecomendaciónAgregar Vitest para pruebas unitarias de businessRules.ts y permissions.ts. Como mínimo, tests de las funciones de cálculo financiero.
